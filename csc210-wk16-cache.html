Part 1. The Memory Hierarchy Problem

The Fundamental Mismatch

Your CPU can execute an instruction in 1 nanosecond (at 1 GHz).

Accessing main RAM takes approximately 100 nanoseconds.

That's 100× slower than the CPU!

The Problem: If pipelining enables one instruction per cycle, but memory accesses take 100 cycles, the CPU spends 99 out of 100 cycles waiting for memory. Pipelining doesn't help—memory is the bottleneck.

This is why every modern CPU has cache memory. A cache is a small, fast memory that stores copies of recently-accessed data from main RAM.

The Memory Hierarchy: Multiple Levels

Rather than one big, slow memory, modern systems use a hierarchy:

┌──────────────────────────────────┐ │ Registers (8-64 bytes) │ Speed: 1 ns (CPU speed) └──────────────────────────────────┘ ↓ ┌──────────────────────────────────┐ │ L1 Cache (32-64 KB) │ Speed: 4 ns (10-100× faster than L2) └──────────────────────────────────┘ ↓ ┌──────────────────────────────────┐ │ L2 Cache (256-512 KB) │ Speed: 10 ns (10× faster than L3) └──────────────────────────────────┘ ↓ ┌──────────────────────────────────┐ │ L3 Cache (8-16 MB) │ Speed: 50 ns (2× faster than RAM) └──────────────────────────────────┘ ↓ ┌──────────────────────────────────┐ │ Main Memory (8-64 GB) │ Speed: 100 ns (baseline) └──────────────────────────────────┘ ↓ ┌──────────────────────────────────┐ │ Storage (SSD/HDD) │ Speed: 1-10 ms (very slow!) └──────────────────────────────────┘

The principle: Small and fast near the CPU, large and slow away from it.

Why Caches Work: Locality

Temporal Locality: If you access data now, you'll likely access it again soon.

for (int i = 0; i < 1000000; i++) { sum += array[i]; // 'sum' accessed 1,000,000 times } // 'sum' stays in cache (extremely hot data)

Spatial Locality: If you access address X, you'll likely access nearby addresses soon.

for (int i = 0; i < 1000; i++) { array[i] = array[i] * 2; // Sequential access } // Addresses accessed in order: excellent spatial locality

These patterns are ubiquitous in real programs. Research shows ~90-95% of memory accesses exhibit temporal or spatial locality.

The Cache Bet: "Most programs access the same small set of data repeatedly." By keeping recently-accessed data in fast cache, we exploit this pattern to achieve near-register speeds without the cost of huge register files.
Part 2: How Caches Work

Cache Operation: Hit or Miss

When the CPU requests data at address X:

1. Check if X is in cache (cache hit) → return immediately (~4 ns) ✓
2. If not found (cache miss) → fetch from main RAM (~100 ns) ✗
3. Store fetched data in cache (replacing old data if necessary)
Hit Rate = Hits ÷ (Hits + Misses)

Effective Memory Access Time (EMAT) =
(Hit Rate × Cache Hit Time) + (Miss Rate × Cache Miss Time)

Example:

100 memory accesses, 95 hit cache, 5 miss Hit Rate = 95 ÷ 100 = 95% Miss Rate = 5 ÷ 100 = 5% EMAT = (0.95 × 4 ns) + (0.05 × 100 ns) = 3.8 ns + 5.0 ns = 8.8 ns Compare to: Always hitting cache: 4 ns Always missing cache: 100 ns Our design: 8.8 ns (much better!)

Cache Organization: Direct-Mapped vs. Associative

Direct-Mapped Cache: Each memory address maps to exactly one cache location.

Address 0x0045 → Cache Entry 5 Address 0x008D → Cache Entry 5 (CONFLICT! Both can't be cached) Problem: "Conflict misses" when multiple addresses map to same location

Set-Associative Cache: Each address maps to one of N possible locations.

2-way associative: Address 0x0045 → Can use Entry 5a or 5b Address 0x008D → Can use Entry 5a or 5b (if one is free) Advantage: Reduces conflict misses

Fully-Associative Cache: Any address can use any location.

Any address can go to any of 16 entries Advantage: No conflict misses Disadvantage: Must search all entries (slow, expensive)

Modern CPUs use set-associative caches (typically 4-8 way) as a middle ground.

Real Cache Configurations

CPU L1 Data L1 Inst. L2 L3
ARM Cortex-A72 32 KB, 4-way 48 KB, 3-way 512 KB 4 MB
Intel i7-8700K 32 KB, 8-way 32 KB, 8-way 256 KB 12 MB
Apple M1 64 KB, 8-way 192 KB, 8-way 2 MB 16 MB

Observation: L1 is small and simple, L3 is large and complex. Design follows: fast/small near CPU, slow/large away from CPU.

The Three Types of Cache Misses

1 Compulsory Miss

First access to a data block (unavoidable). Cannot optimize away.

2 Capacity Miss

Cache too small to hold all data in working set. Solution: larger cache (but slower and more expensive).

3 Conflict Miss

Multiple addresses map to same cache location. Solution: increase associativity (but more complex hardware).

Part 5: Real CPU Cache Hierarchies

ARM Cortex-A72: Balanced Design

L1 Caches: 32 KB data + 48 KB instruction (separate for parallel access)

L2 Cache: 512 KB per core (larger than L1, shared by all core activities)

L3 Cache: 4 MB shared by all cores (helps multi-threaded workloads)

Design Philosophy: Separate I-cache and D-cache in L1 (allows simultaneous fetch and memory access). L2/L3 unified (any data can go there).

Intel Core i9: Maximizing L3 for Performance

L1 Caches: 32 KB each (instruction and data)

L2 Cache: 256 KB per core (victim cache for L1)

L3 Cache: 12-20 MB shared (very large for aggressive prefetching)

Design Philosophy: Aggressive L3 sizing. Large L3 helps with working set misses and enables sophisticated prefetching algorithms.

Apple M1: Specialized Caching

L1 Caches: 64 KB data + 192 KB instruction (larger L1, exploits spatial locality)

L2 Cache: 2 MB per core (much larger than typical)

L3 Cache: 16 MB shared (largest in this generation)

Design Philosophy: Larger L1 (especially L1I) to reduce trips to L2. Combined with aggressive prefetching, reduces memory pressure.

Summary: Why Cache Matters

Without caches, memory is the bottleneck (100× slower than CPU). With effective caching:

Modern CPUs achieve 8-10 ns effective memory latency (compared to 100 ns for uncached RAM)
Hit rates typically 90-95% for well-written code
EMAT of ~10 ns enables pipelining to work (multiple instructions in flight)
Code optimization for cache (sequential access, loop blocking) can provide 5-10× speedup
